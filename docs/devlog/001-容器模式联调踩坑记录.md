# VigilClaw 容器模式端到端联调踩坑记录

> 日期：2026-03-12 | 阶段：Phase 1 E2E 测试

---

## 背景

Phase 1 代码完成后，进行 Telegram → Docker Container → Claude API → 回复的端到端联调。从第一条消息发出到容器模式完全跑通，经历了 7 个坑点，耗时约 3 小时。

本文记录每个问题的现象、根因、修复方案，供后续开发者避坑。

---

## 坑点 1：Docker Socket 未连接

**现象**：发送消息后返回 `⚠️ 任务执行失败: connect ENOENT /var/run/docker.sock`

**根因**：macOS 上 Docker Desktop / OrbStack 未启动，`/var/run/docker.sock` 不存在。

**修复**：启动 Docker Desktop。代码层面增加了 `LocalRunner` 作为降级方案，`VIGILCLAW_LOCAL_MODE=true` 跳过容器直接在宿主机进程内调 API。

**预防**：启动时检测 Docker 可用性（`docker.ping()`），不可用时自动降级并打印 warn 日志。

---

## 坑点 2：Docker 挂载路径必须是绝对路径

**现象**：`(HTTP code 400) bad parameter - includes invalid characters for a local volume name`

**根因**：`prepareIpcDir()` 返回的是相对路径 `data/ipc/<taskId>`，Docker Binds 要求绝对路径，相对路径被当成 named volume 名解析。

**修复**：`ipc.ts` 中 `path.join` → `path.resolve`，一行改动。

```typescript
// Before
const ipcDir = path.join(dataDir, 'ipc', taskId);
// After
const ipcDir = path.resolve(dataDir, 'ipc', taskId);
```

**预防**：所有传给 Docker API 的路径都用 `path.resolve()` 确保绝对路径。

---

## 坑点 3：自定义 seccomp profile 导致容器段错误

**现象**：容器退出码 139（SIGSEGV），日志 `Assertion failed: errno == EINTR (../deps/uv/src/unix/linux.c: uv__io_poll)`

**根因**：自定义 seccomp profile 的系统调用白名单遗漏了 libuv/Node.js 运行时需要的某些 syscall（可能是 `epoll_pwait2` 或 `io_uring` 相关），导致 libuv 的 I/O 轮询触发 SIGSEGV。

**修复**：临时禁用自定义 seccomp，使用 Docker 默认 seccomp profile（已经阻止了 44 个危险 syscall，足够安全）。

```typescript
// Before
const securityOpt = ['no-new-privileges:true', `seccomp=${JSON.stringify(profile)}`];
// After
const securityOpt = ['no-new-privileges:true'];
```

**后续**：需要在 Linux 环境（而非 macOS Docker）上逐步调试 seccomp profile，用 `strace` 记录 Node.js 实际使用的 syscall 列表。

## 坑点 4：Telegram Markdown 解析失败导致回复静默丢失

**现象**：容器报错了，但 Telegram 上没收到任何回复，看起来像"卡住"。

**根因**：错误消息里包含 Docker 日志的控制字符（`\u0002` 等二进制前缀），grammY 用 `parse_mode: 'Markdown'` 发送时 Telegram API 返回 `400: can't parse entities`，回复发送失败。

**修复**：`telegram.ts` 的 `sendMessage` 增加降级逻辑 — Markdown 发送失败时自动用纯文本重试。

```typescript
try {
  await this.bot.api.sendMessage(chatId, chunk, { parse_mode: 'Markdown' });
} catch {
  await this.bot.api.sendMessage(chatId, chunk);
}
```

**预防**：所有发给 Telegram 的文本都应做 Markdown 安全转义，或默认用纯文本。

---

## 坑点 5：AutoRemove 导致容器日志丢失

**现象**：容器退出后 `docker logs` 返回 `No such container`，无法调试。

**根因**：`HostConfig.AutoRemove: true` 使容器退出后立即删除，包括日志。

**修复**：改为 `AutoRemove: false`，在 `container-runner.ts` 的 `finally` 块中手动获取日志后再 `container.remove()`。同时用 `container.wait()` 和 `Promise.race` 竞争结果文件和容器退出，确保两种情况都能捕获。

```typescript
const result = await Promise.race([
  waitForResult(ipcDir, taskId, timeout),
  container.wait().then(async (waitResult) => {
    const logs = await container.logs({ stdout: true, stderr: true });
    // 处理日志和退出码
  }),
]);
```

## 坑点 6：Credential Proxy URL 拼接丢失路径前缀

**现象**：中转服务返回 404，但直接用 Anthropic SDK（LocalRunner 模式）正常。

**根因**：`new URL('/v1/messages', 'https://api-proxy.example.com/anthropic')` 的 `URL` 构造函数行为 — 绝对路径 `/v1/messages` 会**覆盖** base URL 的路径部分 `/anthropic`，结果是 `https://api-proxy.example.com/v1/messages` 而非预期的 `https://api-proxy.example.com/anthropic/v1/messages`。

**修复**：字符串拼接代替 URL 构造函数。

```typescript
// Before (WRONG)
const targetUrl = new URL(requestPath, targetBase);
// After (CORRECT)
const base = targetBase.endsWith('/') ? targetBase.slice(0, -1) : targetBase;
const targetUrl = new URL(base + requestPath);
```

**预防**：永远不要用 `new URL(path, base)` 拼接带路径前缀的 base URL。这是 WHATWG URL 标准的行为，不是 bug。

---

## 坑点 7：macOS Docker 不支持 Unix Socket 卷挂载共享

**现象**：容器退出码 1，日志为空，Agent Runner 报 `Connection error`。

**根因**：技术方案设计了 `NetworkMode: none` + Unix Socket 的零网络方案（方案 E）。宿主机 Credential Proxy 创建 Unix Socket 文件，通过卷挂载共享给容器。但 macOS 上 Docker（OrbStack/Docker Desktop）运行在 Linux VM 中，Unix Socket 文件通过 VirtioFS/gRPC-FUSE 挂载时**不支持跨 VM 的 socket 连接**。容器内看到 socket 文件存在但无法 `connect()`。

**修复**：放弃 Unix Socket 方案，改为 TCP 端口方案。Credential Proxy 每个任务监听一个随机 TCP 端口（`listen(0, '127.0.0.1')`），容器通过 `host.docker.internal:<port>` 访问宿主机。

```typescript
// Credential Proxy: 监听随机端口
server.listen(0, '127.0.0.1', () => {
  const port = (server.address() as AddressInfo).port;
});

// Container Runner: 传端口给容器
Env: [`CREDENTIAL_PROXY_URL=http://host.docker.internal:${port}`];

// Agent Runner: 直接用 HTTP
new Anthropic({ baseURL: process.env.CREDENTIAL_PROXY_URL, apiKey: 'proxy-injected' });
```

**安全影响**：容器不再是 `NetworkMode: none`（需要网络访问 `host.docker.internal`）。API Key 仍然不进容器（由 Proxy 注入），但容器理论上可以扫描宿主机端口。后续需要加 iptables 规则限制容器只能访问 Proxy 端口。

**Linux 环境下**：Unix Socket 方案应该可以工作（无 VM 隔离层），可以在 Linux 部署时切回 `NetworkMode: none` + Unix Socket 方案。

---

## .env 文件格式陷阱

**现象**：`VIGILCLAW_LOCAL_MODE=true` 环境变量读不到。

**根因**：`echo "VIGILCLAW_LOCAL_MODE=true" >> .env` 追加时，如果前一行末尾没有换行符，新变量会粘到上一个变量的值末尾。`ANTHROPIC_AUTH_TOKEN=xxx...kVIGILCLAW_LOCAL_MODE=true` 变成了一个值。

**预防**：写 `.env` 文件时确保每行以 `\n` 结尾。或者用 `printf '\nKEY=value\n' >> .env`。

## 总结

| #    | 类别         | 耗时  | 可预防                         |
| ---- | ------------ | ----- | ------------------------------ |
| 1    | 环境         | 5min  | ✅ 启动检查                    |
| 2    | Docker API   | 10min | ✅ 统一用 `path.resolve`       |
| 3    | 安全配置     | 20min | ⚠️ 需要 Linux 环境测试         |
| 4    | Telegram API | 15min | ✅ 默认纯文本或自动降级        |
| 5    | 调试体验     | 15min | ✅ 禁用 AutoRemove             |
| 6    | URL 标准     | 30min | ✅ 不用 `new URL(path, base)`  |
| 7    | 跨平台       | 60min | ⚠️ macOS Docker ≠ Linux Docker |
| .env | 格式         | 5min  | ✅ 确保换行                    |

**最大教训**：macOS Docker 和 Linux Docker 在文件系统层面行为不同（VM 隔离层），涉及 Unix Socket、文件监控（inotify vs FSEvents）等底层机制时需要特别注意。技术方案中的"方案 E（零网络 + Unix Socket）"在 macOS 开发环境下不可用，需要 TCP 降级方案。
