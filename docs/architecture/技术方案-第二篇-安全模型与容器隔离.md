## 2.1 威胁模型

### 2.1.1 核心假设

VigilClaw 的安全模型建立在一个前提上：

> **Agent 进程是不可信的。** 无论提示词工程多完善，都必须假设 Agent 可能因 Prompt Injection 被攻击者控制，执行任意恶意行为。

这个假设决定了所有安全设计的方向 —— 安全边界不是 "阻止 Agent 做坏事"，而是 "即使 Agent 做了坏事，损害也被控制在容器内"。

### 2.1.2 攻击面分析

| 攻击向量             | 攻击路径                                            | 严重程度 | 我们的防御                                |
| -------------------- | --------------------------------------------------- | -------- | ----------------------------------------- |
| **Prompt Injection** | 用户消息/外部内容注入恶意指令 → Agent 执行任意命令  | 🔴 高    | 容器隔离：命令只在沙箱内生效              |
| **数据外泄**         | Agent 读取工作目录文件 → 通过网络发送到攻击者服务器 | 🔴 高    | 网络白名单：仅允许出站到 LLM API          |
| **API Key 窃取**     | Agent 执行 `printenv`/读取配置 → 获取明文密钥       | 🔴 高    | Credential Proxy：Key 不进容器            |
| **容器逃逸**         | 利用内核漏洞从容器突破到宿主机                      | 🔴 高    | seccomp + no-new-privileges + CapDrop ALL |
| **内网扫描**         | Agent 扫描 `192.168.x.x` / `10.x.x.x`               | 🟠 中    | iptables 阻断 RFC 1918 地址段             |
| **供应链攻击**       | 恶意 npm 依赖注入后门到宿主机进程                   | 🟠 中    | 依赖最小化 (<50 包) + lockfile + audit    |
| **DDoS / 消息洪水**  | 大量消息耗尽容器资源或 API 预算                     | 🟠 中    | 速率限制 + 预算控制                       |
| **宿主机进程攻击**   | 通过 IPC 注入恶意数据攻击宿主机 JSON 解析           | 🟡 低    | Zod Schema 校验所有 IPC 输入              |

### 2.1.4 不防御的场景

| 场景                | 原因                                     |
| ------------------- | ---------------------------------------- |
| 用户自身是攻击者    | 自托管场景，用户即管理员，有物理访问权限 |
| 物理访问攻击        | 超出软件防御范围                         |
| LLM Provider 被攻陷 | 依赖 Anthropic 的安全保障，我们无法防御  |
| 0-day 内核漏洞      | 需要 gVisor/Kata 级别隔离，Phase 3 考虑  |

## 2.2 容器隔离方案

### 2.2.1 Docker 安全配置全景

每个 Agent 容器的完整安全参数：

```typescript
const CONTAINER_SECURITY_CONFIG: Docker.ContainerCreateOptions['HostConfig'] = {
  // ---- 权限控制 ----
  SecurityOpt: [
    'no-new-privileges:true', // 禁止进程获取新权限 (setuid 等)
  ],
  CapDrop: ['ALL'], // 移除所有 Linux Capabilities
  // 不需要 CapAdd —— Agent 不需要任何特权操作
  Privileged: false, // 禁止特权模式（显式声明）

  // ---- 文件系统 ----
  ReadonlyRootfs: true, // 根文件系统只读
  Tmpfs: {
    '/tmp': 'rw,noexec,nosuid,size=100m', // /tmp 可写但不可执行
  },
  // /workspace 和 /ipc 通过 Binds 挂载

  // ---- 资源限制 ----
  Memory: 512 * 1024 * 1024, // 512MB 内存硬限制
  MemorySwap: 512 * 1024 * 1024, // 禁用 swap（与 Memory 相等）
  CpuQuota: 100_000, // 1 核 CPU（100000μs/period）
  CpuPeriod: 100_000,
  PidsLimit: 100, // 最多 100 个进程（防 fork 炸弹）

  // ---- 网络 ----
  NetworkMode: 'vigilclaw-restricted', // 自定义受限网络

  // ---- 其他 ----
  AutoRemove: true, // 退出后自动删除容器
  UsernsMode: '', // 使用默认用户命名空间
};
```

#### 每项配置的安全意义

| 配置                | 防御的攻击          | 如果不设会怎样                                              |
| ------------------- | ------------------- | ----------------------------------------------------------- |
| `no-new-privileges` | SUID/SGID 提权      | 容器内 root 可能通过 setuid 程序提权                        |
| `CapDrop: ALL`      | 特权系统调用        | 默认容器有 14 个 capabilities，包括 `NET_RAW`（可嗅探网络） |
| `ReadonlyRootfs`    | 持久化后门          | 攻击者可修改系统二进制（如 `/usr/bin/curl`）植入后门        |
| `Tmpfs noexec`      | 上传+执行恶意二进制 | 攻击者下载恶意程序到 /tmp 并执行                            |
| `Memory 512MB`      | 内存耗尽 DoS        | 恶意进程可耗尽宿主机内存导致 OOM                            |
| `PidsLimit 100`     | Fork 炸弹           | `:(){ :\|:& };:` 可产生无限进程                             |
| `AutoRemove`        | 容器残留            | 退出的容器保留磁盘空间和潜在的敏感数据                      |

### 2.2.3 卷挂载安全

#### 挂载策略

```typescript
function buildBinds(task: QueuedTask, ipcDir: string): string[] {
  const binds: string[] = [];

  // 1. IPC 通道 — 读写
  binds.push(`${ipcDir}:/ipc:rw`);

  // 2. 工作目录 — 读写（用户指定的项目目录）
  if (task.workspaceDir) {
    // ★ 安全校验：必须是允许的路径
    validateMountPath(task.workspaceDir);
    binds.push(`${task.workspaceDir}:/workspace:rw`);
  }

  return binds;
}
```

#### 挂载白名单校验 (`src/mount-security.ts`)

```typescript
// 绝对禁止挂载的路径（硬编码，不可配置）
const FORBIDDEN_PATHS = [
  '/',
  '/etc',
  '/var',
  '/usr',
  '/bin',
  '/sbin',
  '/lib',
  '/boot',
  '/dev',
  '/proc',
  '/sys',
  '/root',
  '/run',
  process.env.HOME + '/.ssh',
  process.env.HOME + '/.gnupg',
  process.env.HOME + '/.config',
  process.env.HOME + '/.aws',
];

// 允许额外挂载的路径白名单（用户配置）
// 存储在 ~/.config/vigilclaw/mount-allowlist.json
// ★ 此文件在宿主机外部，容器内 Agent 无法修改
interface MountAllowlist {
  paths: string[]; // ["/home/user/projects", "/data/repos"]
}

export function validateMountPath(requestedPath: string): void {
  const resolved = path.resolve(requestedPath);

  // 检查禁止列表
  for (const forbidden of FORBIDDEN_PATHS) {
    if (resolved === forbidden || resolved.startsWith(forbidden + '/')) {
      throw new SecurityError(`Mount path "${resolved}" is forbidden`, 'mount_violation');
    }
  }

  // 检查允许列表
  const allowlist = loadMountAllowlist();
  const isAllowed = allowlist.paths.some(
    (allowed) => resolved === allowed || resolved.startsWith(allowed + '/'),
  );

  if (!isAllowed) {
    throw new SecurityError(`Mount path "${resolved}" is not in allowlist`, 'mount_violation');
  }
}
```

**设计决策**：白名单配置文件放在 `~/.config/vigilclaw/` 而非项目目录内，确保容器内的 Agent 无法通过修改项目文件来绕过挂载限制。这与 NanoClaw 的 `mount-allowlist.json` 方案一致。

### 2.2.4 容器镜像安全

#### Dockerfile（`container/agent-runner/Dockerfile`）

```dockerfile
# ---- 构建阶段 ----
FROM node:22-alpine AS builder
WORKDIR /build
COPY package.json pnpm-lock.yaml ./
RUN corepack enable && pnpm install --frozen-lockfile --prod
COPY src/ ./src/
COPY tsconfig.json ./
RUN pnpm exec tsc

# ---- 运行阶段 ----
FROM node:22-alpine AS runtime

# 安全：创建非 root 用户
RUN addgroup -S agent && adduser -S agent -G agent

# 安全：移除不必要的包
RUN apk del --purge apk-tools && \
    rm -rf /var/cache/apk/* /tmp/* /root/.cache

WORKDIR /app
COPY --from=builder --chown=agent:agent /build/dist/ ./
COPY --from=builder --chown=agent:agent /build/node_modules/ ./node_modules/

# 创建必要的可写目录
RUN mkdir -p /ipc/input /ipc/output /workspace /tmp && \
    chown -R agent:agent /ipc /workspace /tmp

# 安全：切换到非 root 用户
USER agent

# 健康检查（可选）
HEALTHCHECK --interval=30s --timeout=5s \
  CMD [ "node", "-e", "process.exit(0)" ]

ENTRYPOINT ["node", "/app/agent-runner.js"]
```

**镜像安全要点**：

| 措施                      | 目的                                    |
| ------------------------- | --------------------------------------- |
| `node:22-alpine` 基础镜像 | 最小化攻击面（~50MB vs ~350MB node:22） |
| 多阶段构建                | 运行镜像不包含构建工具（tsc, pnpm 等）  |
| 非 root 用户 (`agent`)    | 容器内进程无法修改系统文件              |
| 移除 `apk-tools`          | 防止容器内安装额外软件包                |
| `--frozen-lockfile`       | 确保依赖完全可复现                      |

## 2.3 网络安全策略

**这是 VigilClaw 与 NanoClaw 的核心差异点。**

NanoClaw 的容器网络完全敞开（Issue #458），VigilClaw 必须做到域名级别的出站限制。

### 2.3.1 方案对比

| 方案                                               | 原理                                                          | 优点                   | 缺点                                                                | 推荐度                |
| -------------------------------------------------- | ------------------------------------------------------------- | ---------------------- | ------------------------------------------------------------------- | --------------------- |
| **A: iptables 直接控制**                           | 宿主机 iptables 规则拦截容器出站流量                          | 内核级强制，无法绕过   | 需要 root 权限；不支持域名，只支持 IP                               | ⭐⭐                  |
| **B: DNS 限制代理**                                | 自建 DNS 服务器，仅解析白名单域名                             | 域名级控制             | 可通过 IP 直连绕过                                                  | ⭐⭐                  |
| **C: Docker network + iptables + DNS 组合**        | 三层防御：Docker 隔离网络 + iptables 阻断私网 + DNS 限制      | 域名级 + IP 级双重防御 | 配置复杂                                                            | ⭐⭐⭐⭐              |
| **D: 透明 HTTP 代理**                              | 所有 HTTPS 流量经过宿主机代理，代理检查目标域名               | 精确的域名控制         | 需要 TLS 中间人（MITM），破坏端到端加密                             | ⭐                    |
| **E: NetworkMode: none + Unix Socket**             | 容器零网络，通过挂载的 Unix Socket 文件与宿主机 Proxy 通信    | 最强隔离，零网络攻击面 | ⚠️ **macOS/Windows Docker 不支持**（VM 隔离层无法转发 Socket 连接） | ⭐⭐⭐⭐⭐ (仅 Linux) |
| **F: TCP Credential Proxy + host.docker.internal** | Proxy 监听随机 TCP 端口，容器通过 `host.docker.internal` 访问 | 全平台兼容，实现简洁   | 容器有网络能力，需额外限制出站                                      | ⭐⭐⭐⭐              |

### 2.3.2 推荐方案：分环境策略

经实测验证（见 `docs/devlog/001-容器模式联调踩坑记录.md`），**方案 E 在 macOS/Windows Docker 环境下不可用**。原因：这些平台上 Docker 运行在 Linux VM 中（Docker Desktop 用 HyperKit/Apple Hypervisor，OrbStack 用轻量 VM），宿主机与容器之间的文件共享走虚拟文件系统协议（VirtioFS/gRPC-FUSE），**不支持 Unix Socket 的连接转发** — 容器内能看到 socket 文件但 `connect()` 返回 `ECONNREFUSED`。

因此采用分环境策略：

| 环境                   | 方案                  | 容器网络            | 安全等级               |
| ---------------------- | --------------------- | ------------------- | ---------------------- |
| **Linux 生产**         | 方案 E（Unix Socket） | `NetworkMode: none` | 🟢 最高 — 零网络攻击面 |
| **macOS/Windows 开发** | 方案 F（TCP 端口）    | 默认 bridge         | 🟡 高 — 需额外限制     |

#### 方案 E：Linux 生产环境（零网络 + Unix Socket）

```
容器 (NetworkMode: none — 完全无网络栈)
  │
  │ Agent 通过挂载的 Unix Socket 发送 HTTP 请求
  │ /ipc/proxy.sock → Credential Proxy
  │
  ▼
Credential Proxy (宿主机)
  │
  │ 1. 验证请求路径（白名单）
  │ 2. 注入真实 API Key
  │ 3. 转发到真实 API
  │
  ▼
api.anthropic.com
```

**安全优势**：容器完全没有网络栈。即使 Agent 被 Prompt Injection 攻陷，也无法发起任何网络连接、DNS 查询或端口扫描。唯一通信通道是 Unix Socket，且只能访问 Credential Proxy 暴露的白名单路径。

**实现**：

```typescript
// Credential Proxy 监听 Unix Socket
server.listen(socketPath, () => {
  fs.chmodSync(socketPath, 0o600);
});

// Container Runner 配置
HostConfig: {
  NetworkMode: 'none';
}
Env: [`CREDENTIAL_PROXY_SOCKET=/ipc/proxy.sock`];
Binds: [`${ipcDir}:/ipc:rw`];

// Agent Runner 使用自定义 fetch 连接 Socket
new Anthropic({
  baseURL: 'http://proxy',
  apiKey: 'proxy-injected',
  fetch: socketFetch(process.env.CREDENTIAL_PROXY_SOCKET),
});
```

**前提**：Linux 原生 Docker（容器和宿主机共享内核，Unix Socket 直接可用）。

#### 方案 F：macOS/Windows 开发环境（TCP 端口 + host.docker.internal）

```
容器 (默认 bridge 网络)
  │
  │ Agent 通过 HTTP 请求到 http://host.docker.internal:<port>
  │
  ▼
Credential Proxy (宿主机, 127.0.0.1:<random-port>)
  │
  │ 1. 验证请求路径（白名单）
  │ 2. 注入真实 API Key
  │ 3. 转发到真实 API
  │
  ▼
api.anthropic.com (或自定义中转服务)
```

**实现**：

```typescript
// Credential Proxy 监听随机 TCP 端口
server.listen(0, '127.0.0.1', () => {
  const port = (server.address() as AddressInfo).port;
});

// Container Runner 配置
Env: [`CREDENTIAL_PROXY_URL=http://host.docker.internal:${port}`];
// 不设 NetworkMode: none（需要网络访问宿主机）

// Agent Runner 直接用 HTTP（零额外代码）
new Anthropic({
  baseURL: process.env.CREDENTIAL_PROXY_URL,
  apiKey: 'proxy-injected',
});
```

**安全补偿措施**（方案 F 比方案 E 多出的攻击面需补偿）：

| 额外风险                       | 补偿措施                                                        | 状态      |
| ------------------------------ | --------------------------------------------------------------- | --------- |

```bash
# 在宿主机上以 root 执行

DOCKER_NETWORK="172.30.0.0/24"
CHAIN_NAME="VIGILCLAW"

# ---- 创建自定义链 ----
iptables -N $CHAIN_NAME 2>/dev/null || iptables -F $CHAIN_NAME

# ---- 规则 1: 允许已建立的连接（响应包） ----
iptables -A $CHAIN_NAME -m state --state ESTABLISHED,RELATED -j ACCEPT

# ---- 规则 2: 允许 DNS 解析（到宿主机 DNS 代理） ----
iptables -A $CHAIN_NAME -p udp --dport 53 -d 172.30.0.1 -j ACCEPT
iptables -A $CHAIN_NAME -p tcp --dport 53 -d 172.30.0.1 -j ACCEPT

# ---- 规则 3: 允许出站到 LLM API ----
# Anthropic API (需要定期更新 IP，或使用 ipset)
for ip in $(dig +short api.anthropic.com); do
  iptables -A $CHAIN_NAME -p tcp --dport 443 -d "$ip" -j ACCEPT
done

# ---- 规则 4: 阻断所有 RFC 1918 私有地址 ----
iptables -A $CHAIN_NAME -d 10.0.0.0/8 -j DROP
iptables -A $CHAIN_NAME -d 172.16.0.0/12 -j DROP
iptables -A $CHAIN_NAME -d 192.168.0.0/16 -j DROP

# ---- 规则 5: 阻断 localhost ----
iptables -A $CHAIN_NAME -d 127.0.0.0/8 -j DROP

# ---- 规则 6: 阻断 link-local ----
iptables -A $CHAIN_NAME -d 169.254.0.0/16 -j DROP

# ---- 规则 7: 阻断 IPv6 本地地址 ----
ip6tables -A $CHAIN_NAME -d ::1 -j DROP 2>/dev/null
ip6tables -A $CHAIN_NAME -d fe80::/10 -j DROP 2>/dev/null
ip6tables -A $CHAIN_NAME -d fc00::/7 -j DROP 2>/dev/null

# ---- 规则 8: 拒绝所有其他出站 ----
iptables -A $CHAIN_NAME -j DROP

# ---- 将规则应用到 Docker 网络的 FORWARD 链 ----
iptables -I FORWARD -s $DOCKER_NETWORK -j $CHAIN_NAME

echo "VigilClaw network policy applied."
```

```typescript
// src/dns-proxy.ts （仅方案 C 需要）
import dgram from 'node:dgram';
import dns from 'node:dns';

const ALLOWED_DOMAINS = new Set([
  'api.anthropic.com',
  // Phase 2:
  // 'api.openai.com',
  // 'generativelanguage.googleapis.com',
]);

export function startDnsProxy(listenAddr: string, port: number = 53): void {
  const server = dgram.createSocket('udp4');

  server.on('message', (msg, rinfo) => {
    const domain = parseDnsQuery(msg);

    if (domain && ALLOWED_DOMAINS.has(domain)) {
      // 白名单域名：转发到上游 DNS
      dns.resolve4(domain, (err, addresses) => {
        if (!err && addresses.length > 0) {
          const response = buildDnsResponse(msg, addresses);
          server.send(response, rinfo.port, rinfo.address);
        }
      });
    } else {
      // 非白名单域名：返回 NXDOMAIN
      const nxdomain = buildNxdomainResponse(msg);
      server.send(nxdomain, rinfo.port, rinfo.address);
      logger.warn({ domain, source: rinfo.address }, 'DNS query blocked');
    }
  });

  server.bind(port, listenAddr);
}
```

### 2.4.2 Credential Proxy 实现

```typescript
// src/credential-proxy.ts
import http from 'node:http';
import https from 'node:https';
import net from 'node:net';
import fs from 'node:fs';

// 允许转发的 API 路径白名单
const ALLOWED_PATHS = [
  '/v1/messages', // Anthropic Messages API
  '/v1/complete', // Anthropic Completions API (legacy)
  // Phase 2:
  // '/v1/chat/completions', // OpenAI API
];

// 允许转发的目标主机
const ALLOWED_HOSTS: Record<string, string> = {
  anthropic: 'https://api.anthropic.com',
  // Phase 2:
  // openai: 'https://api.openai.com',
};

export class CredentialProxy {
  private activeSockets = new Map<string, net.Server>(); // taskId → socket server

  constructor(
    private db: Database,
    private masterKey: Buffer,
  ) {}

  /**
   * 为指定任务创建 Unix Socket 代理
   */
  async createProxyForTask(taskId: string, ipcDir: string, provider: string): Promise<string> {
    const socketPath = path.join(ipcDir, 'proxy.sock');

    const server = http.createServer(async (req, res) => {
      try {
        await this.handleRequest(req, res, taskId, provider);
      } catch (err) {
        logger.error({ err, taskId }, 'Credential proxy error');
        res.writeHead(502);
        res.end(JSON.stringify({ error: 'Proxy error' }));
      }
    });

    await new Promise<void>((resolve) => {
      server.listen(socketPath, () => {
        // 设置权限：仅 owner 可读写
        fs.chmodSync(socketPath, 0o600);
        resolve();
      });
    });

    this.activeSockets.set(taskId, server);
    return socketPath;
  }

  private async handleRequest(
    req: http.IncomingMessage,
    res: http.ServerResponse,
    taskId: string,
    provider: string,
  ): Promise<void> {
    // ① 验证请求路径
    const requestPath = req.url ?? '';
    if (!ALLOWED_PATHS.some((p) => requestPath.startsWith(p))) {
      this.logSecurityEvent(taskId, 'credential_access', {
        blocked_path: requestPath,
      });
      res.writeHead(403);
      res.end(JSON.stringify({ error: `Path "${requestPath}" not allowed` }));
      return;
    }

    // ② 获取目标主机
    const targetBase = ALLOWED_HOSTS[provider];
    if (!targetBase) {
      res.writeHead(400);
      res.end(JSON.stringify({ error: `Unknown provider: ${provider}` }));
      return;
    }

    // ③ 解密 API Key
    const apiKey = this.decryptApiKey(provider);

    // ④ 收集请求体
    const bodyChunks: Buffer[] = [];
    for await (const chunk of req) {
      bodyChunks.push(chunk as Buffer);
    }
    const body = Buffer.concat(bodyChunks);

    // ⑤ 转发请求（注入 Authorization）
    const targetUrl = new URL(requestPath, targetBase);

    const proxyReq = https.request(targetUrl, {
      method: req.method,
      headers: {
        ...req.headers,
        host: targetUrl.hostname,
        authorization: `Bearer ${apiKey}`, // ★ 注入真实 Key
        'anthropic-version': '2023-06-01',
        'content-length': body.length.toString(),
      },
    });

    // ⑥ 转发响应
    proxyReq.on('response', (proxyRes) => {
      res.writeHead(proxyRes.statusCode ?? 500, proxyRes.headers);
      proxyRes.pipe(res);

      // ⑦ 异步记录 usage（不阻塞响应）
      this.recordUsage(taskId, provider, proxyRes, body);
    });

    proxyReq.on('error', (err) => {
      logger.error({ err, taskId }, 'Upstream request failed');
      res.writeHead(502);
      res.end(JSON.stringify({ error: 'Upstream request failed' }));
    });

    proxyReq.write(body);
    proxyReq.end();
  }

  /**
   * 关闭指定任务的代理
   */
  async destroyProxyForTask(taskId: string): Promise<void> {
    const server = this.activeSockets.get(taskId);
    if (server) {
      server.close();
      this.activeSockets.delete(taskId);
    }
  }

  /**
   * 解密存储的 API Key
   */
  private decryptApiKey(provider: string): string {
    const cred = this.db.getCredential(provider);
    if (!cred) {
      throw new Error(`No credential found for provider: ${provider}`);
    }

    return decrypt(cred.keyEncrypted, cred.iv, this.masterKey);
  }

  /**
   * 异步记录 API 使用量
   */
  private async recordUsage(
    taskId: string,
    provider: string,
    proxyRes: http.IncomingMessage,
    requestBody: Buffer,
  ): Promise<void> {
    // 从响应中提取 usage 信息（需要缓冲完整响应体）
    // 注意：对于流式响应，usage 在最后一个 SSE 事件中
    // 这里简化处理：从请求体估算输入 token，从响应提取实际 usage
    try {
      const request = JSON.parse(requestBody.toString());
      const model = request.model;
      // usage 将在任务完成时通过 IPC result 获取，这里仅记录请求
      logger.debug({ taskId, model, provider }, 'API request proxied');
    } catch {
      // 非 JSON 请求（如流式），忽略
    }
  }

  private logSecurityEvent(
    taskId: string,
    eventType: string,
    details: Record<string, unknown>,
  ): void {
    this.db.insertSecurityEvent({
      eventType,
      userId: taskId, // 通过 taskId 关联到用户
      details,
      severity: 'medium',
    });
  }
}
```

### 2.4.3 凭证存储加密

#### 加密方案

```typescript
// src/crypto.ts
import crypto from 'node:crypto';

const ALGORITHM = 'aes-256-gcm';
const IV_LENGTH = 16;
const AUTH_TAG_LENGTH = 16;

/**
 * 加密 API Key
 */
export function encrypt(plaintext: string, masterKey: Buffer): { encrypted: Buffer; iv: Buffer } {
  const iv = crypto.randomBytes(IV_LENGTH);
  const cipher = crypto.createCipheriv(ALGORITHM, masterKey, iv);

  const encrypted = Buffer.concat([cipher.update(plaintext, 'utf-8'), cipher.final()]);

  const authTag = cipher.getAuthTag();

  // encrypted = ciphertext + authTag
  return {
    encrypted: Buffer.concat([encrypted, authTag]),
    iv,
  };
}

/**
 * 解密 API Key
 */
export function decrypt(encrypted: Buffer, iv: Buffer, masterKey: Buffer): string {
  const authTag = encrypted.subarray(encrypted.length - AUTH_TAG_LENGTH);
  const ciphertext = encrypted.subarray(0, encrypted.length - AUTH_TAG_LENGTH);

  const decipher = crypto.createDecipheriv(ALGORITHM, masterKey, iv);
  decipher.setAuthTag(authTag);

  return Buffer.concat([decipher.update(ciphertext), decipher.final()]).toString('utf-8');
}
```

#### Master Key 管理

```typescript
/**
 * 首次启动时自动生成 Master Key

export function ensureMasterKey(): Buffer {
  const envKey = process.env.VIGILCLAW_MASTER_KEY;

  if (envKey) {
    // 从环境变量加载（生产推荐）
    if (envKey.length !== 64) {
      throw new Error('VIGILCLAW_MASTER_KEY must be 64 hex characters (32 bytes)');
    }
    return Buffer.from(envKey, 'hex');
  }

  // 自动生成并写入文件（首次启动）
  const keyPath = path.join(os.homedir(), '.config', 'vigilclaw', 'master.key');

  if (fs.existsSync(keyPath)) {
    const hex = fs.readFileSync(keyPath, 'utf-8').trim();
    return Buffer.from(hex, 'hex');
  }

  // 生成新 Key
  const newKey = crypto.randomBytes(32);
  fs.mkdirSync(path.dirname(keyPath), { recursive: true });
  fs.writeFileSync(keyPath, newKey.toString('hex'), { mode: 0o600 });

  logger.warn({ keyPath }, '⚠️  New Master Key generated. BACK UP THIS FILE!');
  return newKey;
}
```

## 2.5 速率限制

### 2.5.1 滑动窗口算法

```typescript
// src/rate-limiter.ts
export class RateLimiter {
  // 内存中的滑动窗口计数器
  private windows = new Map<string, number[]>(); // key → timestamp[]

  constructor(private config: RateLimitConfig) {}

  /**
   * 检查是否超过限制
   * @returns true 如果被限流
   */
  isLimited(userId: string, groupId?: string): boolean {
    const now = Date.now();
    const windowMs = 60_000; // 1 分钟窗口

    // 检查用户级
    if (this.checkWindow(`user:${userId}`, now, windowMs, this.config.perUser)) {
      return true;
    }

    // 检查群组级
    if (groupId && this.checkWindow(`group:${groupId}`, now, windowMs, this.config.perGroup)) {
      return true;
    }

    // 检查全局级
    if (this.checkWindow('global', now, windowMs, this.config.global)) {
      return true;
    }

    return false;
  }

  private checkWindow(key: string, now: number, windowMs: number, limit: number): boolean {
    let timestamps = this.windows.get(key) ?? [];

    // 清除过期的时间戳
    timestamps = timestamps.filter((t) => now - t < windowMs);

    if (timestamps.length >= limit) {
      return true; // 超过限制
    }

    // 记录本次请求
    timestamps.push(now);
    this.windows.set(key, timestamps);
    return false;
  }

  /**
   * 定期清理过期数据（防止内存泄漏）
   */
  cleanup(): void {
    const now = Date.now();
    for (const [key, timestamps] of this.windows) {
      const valid = timestamps.filter((t) => now - t < 60_000);
      if (valid.length === 0) {
        this.windows.delete(key);
      } else {
        this.windows.set(key, valid);
      }
    }
  }
}
```

| **Lockfile 严格模式** | `pnpm install --frozen-lockfile`       | CI 中强制使用            |
| **依赖数量监控**      | CI 脚本检查 `pnpm list --depth=0` 输出 | 超过 50 个阻断构建       |
| **安全审计**          | `pnpm audit --audit-level=high`        | CI 中 High/Critical 阻断 |
| **新增依赖审查**      | PR Review 中 lockfile diff 必须审查    | 人工审查                 |
| **定期更新**          | Dependabot / Renovate 自动 PR          | 每周执行                 |

#### CI 依赖检查脚本

```bash
#!/bin/bash
# scripts/check-deps.sh

set -e

# 检查依赖数量
DEP_COUNT=$(pnpm list --depth=0 --prod --json | node -e "
  const data = JSON.parse(require('fs').readFileSync('/dev/stdin','utf8'));
  console.log(Object.keys(data[0].dependencies || {}).length);
")

echo "Production dependencies: $DEP_COUNT"

if [ "$DEP_COUNT" -gt 50 ]; then
  echo "❌ ERROR: Too many dependencies ($DEP_COUNT > 50)"
  exit 1
fi

# 安全审计
echo "Running security audit..."
pnpm audit --audit-level=high

echo "✅ Dependency check passed."
```

## 2.8 安全模型总结

### 五层防御全景

```
┌───────────────────────────────────────────────────┐
│  Layer 5: 供应链安全                                │
│  lockfile 锁定 → audit 扫描 → 依赖最小化 (<50包)    │
├───────────────────────────────────────────────────┤
│  Layer 4: 速率限制与成本控制                         │
│  用户/群组/全局限流 → 预算上限 → 超限阻断            │
├───────────────────────────────────────────────────┤
│  Layer 3: 凭证零信任                                │
│  Credential Proxy → 加密存储 → 容器内无明文 Key      │
├───────────────────────────────────────────────────┤
│  Layer 2: 网络隔离（分环境策略）                      │
│  Linux:  方案E → NetworkMode:none + Unix Socket    │
│  macOS:  方案F → TCP Proxy + host.docker.internal  │
│  (可叠加: 方案C iptables 白名单限制出站)             │
├───────────────────────────────────────────────────┤
│  Layer 1: 容器隔离                                  │
│  只读rootfs → CapDrop ALL → seccomp → 资源限制      │
│  → no-new-privileges → 非root用户                   │
└───────────────────────────────────────────────────┘
```

### 安全配置默认值汇总

| 配置项             | 默认值                                                    | 可否覆盖                  |
| ------------------ | --------------------------------------------------------- | ------------------------- |
| 容器网络模式       | Linux: `none` (方案E) / macOS: bridge + TCP Proxy (方案F) | 自动检测环境              |
| 容器文件系统       | `ReadonlyRootfs: true`                                    | ❌ 不可覆盖               |
| Linux Capabilities | `CapDrop: ALL`                                            | ❌ 不可覆盖               |
| 凭证存储           | AES-256-GCM 加密                                          | ❌ 不可覆盖               |
| 容器内用户         | `agent` (非 root)                                         | ❌ 不可覆盖               |
| 内存限制           | 512MB                                                     | ✅ 可调整（最大 2GB）     |
| CPU 限制           | 1 核                                                      | ✅ 可调整                 |
| 进程限制           | 100 PID                                                   | ✅ 可调整（最大 500）     |
| 任务超时           | 5 分钟                                                    | ✅ 可调整（最大 30 分钟） |
| 速率限制(用户)     | 10 条/分钟                                                | ✅ 可调整                 |
| 速率限制(群组)     | 30 条/分钟                                                | ✅ 可调整                 |
| 挂载禁止路径       | `/etc`, `/var`, `~/.ssh` 等                               | ❌ 不可覆盖               |
| API 路径白名单     | `/v1/messages`, `/v1/complete`                            | ✅ 可扩展                 |
| 网络出站白名单     | `api.anthropic.com`                                       | ✅ 可扩展                 |
| 日志敏感信息       | 自动脱敏 `sk-ant-*`                                       | ❌ 不可覆盖               |

**设计原则**：安全相关的配置默认值已经是最佳实践，用户只需要扩展（加白名单），不需要也不能削弱。
