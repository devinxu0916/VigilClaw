# VigilClaw 数据库选型决策

> 版本：v1.0 | 日期：2026-03-19 | 关联文档：技术方案第三篇

---

## 决策总览

| 层级 | 选择 | 放弃的选项 |
|------|------|-----------|
| 主数据库 | SQLite (better-sqlite3) | PostgreSQL / MySQL |
| 访问方式 | 直接 SQL + Prepared Statement | Drizzle ORM / Prisma / TypeORM |
| 向量搜索 | sqlite-vec 扩展 | Pinecone / Weaviate / Qdrant |
| 迁移方案 | user_version PRAGMA 自实现 | db-migrate / Flyway / Drizzle Kit |

**一句话总结**：每一层都在"满足当前需求"和"控制复杂度"之间取最保守的点，同时在 DAL 层做足够封装保留未来升级路径。

---

## 第一层：为什么是 SQLite，而不是 PostgreSQL / MySQL

**核心约束驱动**：项目有一个硬性原则——"零外部依赖，数据完全可控"。PostgreSQL 需要独立进程、独立端口、独立配置，docker-compose 里要多一个 service，部署复杂度直接翻倍。SQLite 是一个文件，备份就是 `cp`，迁移就是 `scp`，这对个人自托管场景是决定性优势。

**并发需求不匹配**：PostgreSQL 的核心价值是多进程并发写入和水平扩展。这个项目的并发模型是：同一 group 内串行（GroupQueue 保证），跨 group 并行但数量有限（max=5 容器）。写入压力极低，SQLite 的 WAL 模式（写入不阻塞读取）完全够用，99th percentile 查询 < 50ms。

**目标用户规模**：PRD 明确定位是个人/小团队（2-5 人）自托管，单实例 SQLite 支撑 < 1000 用户没有问题。如果规模增长到需要 PostgreSQL，那时候项目的性质已经变了，是另一个决策点（ROADMAP Phase 3 预留了迁移路径）。

---

## 第二层：为什么是 better-sqlite3，而不是 node-sqlite3 / sql.js

| 决策因素 | better-sqlite3 | node-sqlite3 / sql.js |
|---------|---------------|----------------------|
| API 模型 | **同步** — 与 SQLite 单线程模型天然匹配 | 异步回调 — 增加不必要的事件循环开销 |
| 性能 | ~100 万 ops/sec 简单查询 | ~10 万 ops/sec（异步开销） |
| 代码可读性 | 直接 SQL，调试时可复制到 sqlite3 CLI 验证 | 回调嵌套，调试困难 |
| 事务控制 | `db.transaction(() => {})()` 同步原子操作 | 需要 async/await + try/catch rollback |

**同步 API 的三个好处**：

1. **代码路径更清晰**：DAL 层直接写 SQL，调试时可以把 SQL 复制到 `sqlite3` CLI 直接验行，不需要理解 ORM 生成了什么。对于一个目标"开发者可以完整读懂系统"的项目，这一点很重要。

2. **事务控制更直接**：`db.transaction(() => { ... })()` 这一行就是原子操作，不需要 `await`、不需要 try/catch 包裹 rollback。迁移系统就是这样实现的——所有 pending migration 在一个事务里执行，要么全成功，要么全回滚。

3. **性能差距显著**：`better-sqlite3` 简单查询约 100 万 ops/sec，异步驱动因为事件循环开销约 10 万 ops/sec，差了一个数量级。虽然这个项目的查询量不会触及瓶颈，但没有理由主动引入 10x 的性能损耗。

## 第三层：为什么不用 ORM（Drizzle / Prisma / TypeORM）

**核心结论**：对于目标 < 10,000 行代码的轻量项目，ORM 是负债而非资产。

**依赖膨胀**：Prisma 需要 `@prisma/client` + `prisma`（CLI）+ 生成代码，Drizzle 需要 `drizzle-orm` + `drizzle-kit`。项目生产依赖目标 < 50 个，ORM 一进来就占掉 3-5 个槽位，还带来 schema 定义文件、生成文件等额外维护负担。

**SQL 透明度**：项目有几个非标准聚合查询，比如成本报告：

```sql
SELECT
  model,
  SUM(cost_usd)                     AS total_cost,
  SUM(input_tokens + output_tokens) AS total_tokens,
  COUNT(*)                          AS call_count
FROM api_calls
WHERE user_id = ? AND date(created_at) = date('now')
GROUP BY model
ORDER BY total_cost DESC
```

这类查询用 ORM 写出来反而更啰嗦，而且 ORM 生成的 SQL 不一定会命中 `idx_api_calls_user_date` 这个表达式索引，需要额外 `EXPLAIN QUERY PLAN` 验证。直接写 SQL 更可控。

**迁移系统自己实现更简单**：用 SQLite 内置的 `user_version` PRAGMA 做版本控制，零依赖，整个迁移系统约 30 行代码。Prisma Migrate 或 Drizzle Kit 的迁移系统功能更强，但对这个规模的项目是过度设计。

## 第四层：为什么用 sqlite-vec，而不是独立向量数据库

Phase 2 加入持久化记忆时，面临选择：Pinecone / Weaviate / Qdrant（独立向量数据库）vs sqlite-vec（SQLite 扩展）。

| 维度 | sqlite-vec | Pinecone / Qdrant |
|------|-----------|------------------|
| 运维 | 零（随主进程启动） | 独立进程 / 云服务 |
| 数据主权 | 本地文件，完全可控 | Pinecone 数据出境 |
| 备份 | 同主数据库一个文件 | 独立备份策略 |
| 规模上限 | 千级条目足够 | 百万级以上才有优势 |
| 成本 | 零 | Pinecone 按用量计费 |

选 sqlite-vec 的核心理由：**架构一致性 + 零运维 + 数据不出本地**。个人助手场景的记忆条目数量级是千级，不是百万级，sqlite-vec 的性能完全够用，不需要 HNSW 索引这类为大规模设计的数据结构。

**踩坑**：sqlite-vec 在 ESM 环境下需要 `createRequire` 加载（项目是 `"type": "module"` 的纯 ESM，直接 `require` 报错）：

```typescript
import { createRequire } from 'node:module';
const require = createRequire(import.meta.url);
const sqliteVec = require('sqlite-vec');
```

且如果 sqlite-vec 扩展不可用（某些 Alpine 镜像），会静默降级禁用记忆功能，不影响主流程。

## 第五层：Schema 设计关键决策

### PRAGMA 调优

```typescript
db.pragma('journal_mode = WAL');      // 读写并发，写入不阻塞读取
db.pragma('synchronous = NORMAL');    // WAL 模式下 NORMAL 足够安全且更快
db.pragma('cache_size = -20000');     // 20MB 页面缓存（负号 = KB）
db.pragma('busy_timeout = 5000');     // 锁等待超时 5s，防止写冲突报错
db.pragma('foreign_keys = ON');       // 启用外键约束
db.pragma('secure_delete = ON');      // 删除数据时零填充，防残留
```

WAL 模式需要注意文件膨胀，项目每 5 分钟检查 WAL 文件大小，超过 50MB 时主动 checkpoint：

```typescript
setInterval(() => {
  const stat = fs.statSync(dbPath + '-wal');
  if (stat.size > 50 * 1024 * 1024) {
    db.pragma('wal_checkpoint(RESTART)');
  }
}, 300_000).unref();
```

### 索引策略

| 索引 | 服务的查询场景 | 设计要点 |
|------|-------------|---------|
| `idx_messages_session` | 加载会话上下文（最近 N 条） | `(session_key, created_at DESC)`，每条消息触发一次 |
| `idx_api_calls_user_date` | `/cost` 命令按日聚合 | `(user_id, date(created_at))` 表达式索引，避免全表扫描 |
| `idx_api_calls_task` | 任务完成时汇总费用 | `(task_id)`，每个任务一次 |
| `idx_tasks_status` | 调度器查找 pending 任务 | `(status, created_at)`，每 30 秒一次 |
| `idx_scheduled_next` | 查找到期定时任务 | `(enabled, next_run_at)`，每 30 秒一次 |
| `idx_security_severity` | 安全事件查询 | `(severity, created_at DESC)`，极低频管理员查询 |

`idx_api_calls_user_date` 是专门为 `/cost` 命令设计的表达式索引。`date(created_at)` 是函数调用，普通索引不会命中，必须显式建表达式索引。如果数据量增长到几十万条，没有这个索引的聚合查询会慢 10-100 倍。

### 时间字段用 TEXT 而不是 INTEGER

SQLite 没有原生 DATETIME 类型，常见两种做法：ISO 8601 字符串（TEXT）或 Unix 时间戳（INTEGER）。选 TEXT 的原因：SQLite 内置的 `date()`、`strftime()`、`datetime()` 函数直接支持 ISO 8601 字符串，成本聚合查询里的 `date(created_at) = date('now')` 和 `strftime('%Y-%m', created_at)` 可以直接用，不需要额外转换。

### credentials 表的 IV 独立存储

每个凭证有独立的 `iv` 字段（初始化向量），而不是共用固定 IV。这是 AES-256-GCM 的安全要求——相同 Key + 相同 IV 加密不同数据，会泄露明文的 XOR 关系。每次写入凭证时生成新的随机 IV，存在同一行里，解密时取出来用。

### ON CONFLICT 语法减少往返

`getOrCreateUser` 用了 SQLite 3.24+ 的 `ON CONFLICT ... DO UPDATE ... RETURNING *` 语法：

```sql
INSERT INTO users (id, name) VALUES (?, ?)
ON CONFLICT(id) DO UPDATE SET name = excluded.name
RETURNING *
```

一条语句完成"不存在则插入，存在则更新名字，返回最终行"，避免了先 SELECT 再 INSERT/UPDATE 的两次往返。

### CASE WHEN 状态机更新

`updateTaskStatus` 用 `CASE WHEN` 在单条 UPDATE 里处理 `started_at` 和 `completed_at` 的条件赋值：

```sql
SET started_at   = CASE WHEN ? = 'running'
                        THEN datetime('now') ELSE started_at END,
    completed_at = CASE WHEN ? IN ('completed','failed','timeout')
                        THEN datetime('now') ELSE completed_at END
WHERE id = ?
```

任务状态机的所有转换（pending→running、running→completed/failed/timeout）用同一条 Prepared Statement 处理，DAL 接口更简洁。

## 第六层：Prepared Statements 的性能意义

DAL 层所有高频查询都用 Prepared Statement，而不是每次拼接 SQL 字符串：

```typescript
// 启动时一次性 prepare，之后复用
private stmts = {
  insertMessage:    this.db.prepare(`INSERT INTO messages ...`),
  getRecentMessages: this.db.prepare(`SELECT ... WHERE session_key = ? LIMIT ?`),
  getUserDayCost:   this.db.prepare(`SELECT COALESCE(SUM(cost_usd), 0) ...`),
  // ...
};

// 调用时直接 .run() / .all() / .get()，不重新解析 SQL
this.stmts.insertMessage.run(sessionKey, userId, groupId, role, content, tokens);
```

两个好处：
1. **性能**：SQLite 只解析和编译一次 SQL，后续调用直接执行，对高频写入（每条消息都要 `insertMessage`）有明显提升
2. **安全**：参数通过 `?` 占位符绑定，天然防 SQL 注入，不会被用户输入污染

## 第七层：迁移系统设计

用 SQLite 内置的 `user_version` PRAGMA 做版本控制，零依赖，整个迁移系统约 30 行：

```typescript
const MIGRATIONS: Migration[] = [
  { version: 1, description: 'Initial schema', up: SCHEMA_V1 },
  { version: 2, description: 'Context compression and persistent memory', up: SCHEMA_V2 },
  { version: 3, description: 'Skills table', up: SCHEMA_V3 },
];

function runMigrations(db: Database.Database): void {
  const currentVersion = db.pragma('user_version', { simple: true }) as number;
  const pending = MIGRATIONS.filter((m) => m.version > currentVersion);
  if (pending.length === 0) return;

  // 所有 pending migration 在一个事务里原子执行
  db.transaction(() => {
    for (const migration of pending) {
      db.exec(migration.up);
      db.pragma(`user_version = ${migration.version}`);
    }
  })();
}
```

`user_version` 是 SQLite 内置的整数字段，专门为应用层版本控制设计，不会和业务数据混在一起。每次追加新 migration 只需在数组里加一个对象，启动时自动检测并执行，已有数据完全不受影响。

## 第八层：数据清理策略

数据不能无限增长，项目设计了自动清理（每天凌晨执行）：

| 表 | 保留时长 | 原因 |
|----|---------|------|
| `messages` | 90 天 | 上下文窗口最多 50 条，历史消息只用于统计，90 天够用 |
| `api_calls` | 365 天 | 成本追踪需要年度视图，超过一年的数据价值极低 |
| `tasks` | 365 天 | 同上 |
| `security_events` | 365 天 | 安全审计需要较长保留期 |
| `context_summaries` | 随 `/clear` 命令同步删除 | 不需要定期清理 |
| `memories` | 手动管理 | 用户主动记忆，不自动过期 |

清理用单条 `DELETE WHERE created_at < datetime('now', '-90 days')` 完成，SQLite WAL 模式下大批量删除不会阻塞读操作。

## 决策树总结

```
需要外部数据库服务吗？
  → 不需要（个人自托管，零运维优先）
    → SQLite

需要 ORM 吗？
  → 不需要（代码量目标 < 10,000 行，SQL 透明度更重要）
    → better-sqlite3 直接调用 + Prepared Statements

需要独立向量数据库吗？
  → 不需要（记忆条目千级规模，数据不出本地）
    → sqlite-vec 扩展

需要专业迁移工具吗？
  → 不需要（user_version PRAGMA 30 行搞定）
    → 自实现迁移系统

未来规模增长怎么办？
  → DAL 层已封装，替换底层实现不影响业务代码
  → ROADMAP Phase 3 预留 PostgreSQL 迁移路径
```
