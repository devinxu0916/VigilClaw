## 5.1 记忆系统总体架构

### 5.1.1 设计目标

个人 AI 助手最核心的体验差异在于**是否能记住用户**。VigilClaw 的持久化记忆系统解决三个层次的问题：

| 层次       | 问题                           | 解决方案           | 数据保留期 |
| ---------- | ------------------------------ | ------------------ | ---------- |
| 短期记忆   | 当前对话上下文如何传递         | 消息历史表         | 90 天      |
| 中期记忆   | 长对话的历史消息如何不丢失     | 上下文压缩（摘要） | 跟随会话   |
| 长期记忆   | 跨会话的用户偏好如何持续生效   | 向量语义记忆       | 365 天     |

### 5.1.2 三层记忆架构

```
                    ┌──────────────────────────────────────┐
                    │           SessionManager             │
                    │         (上下文组装 · 协调者)          │
                    └──────┬──────────┬──────────┬─────────┘
                           │          │          │
                ┌──────────▼──┐ ┌─────▼──────┐ ┌─▼──────────────┐
                │  短期记忆    │ │  中期记忆   │ │    长期记忆     │
                │ messages 表  │ │ 上下文压缩  │ │  向量语义记忆   │
                │  (最近 N 条) │ │  (摘要合并) │ │ (语义检索事实)  │
                └─────────────┘ └────────────┘ └────────────────┘
```

每次用户发消息时，`SessionManager.getContext()` 按以下顺序组装完整上下文：

1. **获取近期消息** → 从 `messages` 表加载最近 N 条
2. **压缩旧消息** → 若 token 数超阈值，`ContextCompressor` 将旧消息摘要化
3. **召回相关记忆** → `MemoryStore.recall()` 用向量检索匹配的长期记忆

最终组装的消息数组结构：

```
[Conversation Summary]  ← 中期记忆（摘要，system role）
[Relevant Memories]     ← 长期记忆（匹配的事实，system role）
user: ...               ← 短期记忆（近期消息原文）
assistant: ...
user: ...（最新一条）
```

## 5.2 短期记忆：消息历史

### 5.2.1 存储与加载

最直接的上下文方式：将每轮对话的 user / assistant 消息存入 `messages` 表，下次对话时按 `session_key` 取出最近 N 条。

```typescript
// src/session-manager.ts
const rows = this.db.getRecentMessages(sessionKey, this.contextLength);
let messages: Message[] = rows.map((r) => ({
  role: r.role as Message['role'],
  content: r.content,
}));
```

| 参数             | 默认值 | 说明                             |
| ---------------- | ------ | -------------------------------- |
| `contextLength`  | 50     | 最多加载消息条数                 |
| `session_key`    | —      | `groupId ?? userId`，会话唯一标识 |

### 5.2.2 会话隔离

```typescript
const sessionKey = groupId ?? userId;
```

- **私聊**：`sessionKey = userId`（如 `telegram:123456`）
- **群聊**：`sessionKey = groupId`（如 `telegram:-100xxxx`）

每个会话的消息互不干扰，群聊中的所有用户共享同一上下文窗口。

### 5.2.3 自动清理

消息保留 90 天，由定时任务每日 UTC 03:00 执行：

```sql
DELETE FROM messages WHERE created_at < datetime('now', '-90 days');
```

## 5.3 中期记忆：上下文压缩

### 5.3.1 为什么需要压缩

LLM 的上下文窗口虽然越来越大，但每条消息都消耗 token = 消耗成本。一个 50 条消息的对话可能超过 10,000 tokens，其中大量是早期不再直接相关的内容。上下文压缩在**不丢失信息**的前提下大幅降低 token 消耗。

### 5.3.2 压缩触发条件

```typescript
// src/context-compressor.ts
const totalTokens = estimateMessagesTokens(messages);
if (totalTokens <= this.config.maxContextTokens) {
  // 未超阈值 → 不压缩，直接返回
  return messages;
}
```

| 参数                 | 默认值 | 说明                     |
| -------------------- | ------ | ------------------------ |
| `maxContextTokens`   | 6000   | token 数超过此值触发压缩 |
| `recentMessagesKeep` | 6      | 压缩时保留最近 N 条原文  |

### 5.3.3 压缩流程

```
消息总量 > 6000 tokens?
  │
  ├─ 否 → 原样返回（如有旧摘要则拼接）
  │
  └─ 是 → 分割消息
           ├── 旧消息（前 N-6 条）→ 送 Claude Haiku 生成摘要
           └── 近期消息（最后 6 条）→ 保留原文
```

生成的摘要存入 `context_summaries` 表，后续对话中合并使用。

### 5.3.4 增量摘要合并

关键设计：**摘要不是每次重新生成的，而是增量合并的**。

```typescript
// 当已有旧摘要时，使用增量 prompt
if (existingSummary) {
  userPrompt = INCREMENTAL_SUMMARY_PROMPT
    .replace('{existing_summary}', existingSummary)
    .replace('{new_messages}', formattedMessages);
}
```

增量策略的优势：

- 旧摘要已经浓缩了更早的对话，不需要重复处理
- 每次只需 summarize 新增的几条消息
- 摘要质量随对话延伸保持稳定，不会因为信息量增大而丢失早期细节

### 5.3.5 摘要 Prompt 设计

```text
You are a conversation summarizer. Your job is to create a concise but thorough summary.

Rules:
- Preserve ALL code snippets, file paths, function names, and technical details exactly
- Preserve user preferences, decisions made, and action items
- Preserve error messages and their resolutions
- Use bullet points for clarity
- Keep the summary under 500 words
- Write in the same language the conversation uses
```

使用 `claude-haiku-4-5-20251001` 模型，`temperature: 0.3`，`maxTokens: 1024`。每次摘要调用成本约 $0.001–0.005。

### 5.3.6 降级策略

如果摘要生成失败（网络异常、API 错误等），降级为**截断保留最近消息**：

```typescript
catch (err) {
  logger.warn({ err, sessionKey }, 'Summarization failed, falling back to truncation');
  return recentMessages; // 只保留最近 6 条
}
```

## 5.4 长期记忆：向量语义记忆

### 5.4.1 核心思想

长期记忆解决的是**跨会话记忆**问题：用户在 3 天前说过"我喜欢用 Vim"，3 天后讨论编辑器时应该记得这个偏好。

实现方式：从每次对话中提取有价值的"事实"（facts），转为向量存储。下次对话时，用用户当前的消息做语义检索，找出相关的历史事实注入上下文。

### 5.4.2 技术栈

长期记忆依赖三个组件协同工作：

```
@huggingface/transformers    sqlite-vec              MemoryStore
(本地 AI 推理引擎)           (向量搜索引擎)          (业务协调层)
        │                        │                       │
        │  加载 all-MiniLM-L6-v2 │                       │
        │  模型                   │                       │
        ▼                        ▼                       ▼
   文本 → 384维向量          向量存储与KNN搜索      提取/存储/检索/去重
```

#### (1) @huggingface/transformers — 本地 AI 推理引擎

Hugging Face 的 JavaScript/TypeScript 推理库，**在本地运行 AI 模型**而非调用远程 API。基于 ONNX Runtime，支持在 Node.js 中直接加载和运行 transformer 模型。

核心价值：**零 API 成本的文本向量化**。每次 embedding 操作没有网络延迟，没有 API 费用。

#### (2) all-MiniLM-L6-v2 — 语义嵌入模型

微软开源的轻量级句子嵌入模型（约 23MB ONNX 格式）。将任意文本映射为 **384 维浮点向量**，语义相近的文本在向量空间中距离更近。

| 属性       | 值                |
| ---------- | ----------------- |
| 输出维度   | 384               |
| 模型大小   | ~23MB (ONNX)      |
| 推理速度   | ~10-50ms/句（CPU）|
| 多语言支持 | 英语为主，中文可用 |

#### (3) sqlite-vec — SQLite 向量搜索扩展

Alex Garcia 开发的 SQLite 扩展，为 SQLite 添加向量存储与 KNN 搜索能力。通过 `vec0` 虚拟表存储向量，使用 `embedding MATCH ?` 语法做余弦距离搜索。

```sql
-- 创建向量表
CREATE VIRTUAL TABLE IF NOT EXISTS vec_memories USING vec0(
  embedding float[384]
);

-- KNN 搜索
SELECT rowid, distance
FROM vec_memories
WHERE embedding MATCH ?
ORDER BY distance
LIMIT ?;
```

#### (4) Embedder — 嵌入模型封装层

```typescript
// src/embedder.ts
export class Embedder {
  private modelName: string;

  constructor(modelName = 'Xenova/all-MiniLM-L6-v2') {
    this.modelName = modelName;
  }

  get available(): boolean {
    return !loadFailed;
  }

  async embed(text: string): Promise<Float32Array> {
    const pipe = await this.getPipeline();
    const result = await pipe(text, { pooling: 'mean', normalize: true });
    return result.data;
  }
}
```

关键设计：

- **懒加载**：首次调用 `embed()` 时才加载模型（约 2-3 秒），避免启动时阻塞
- **单例模式**：`pipelineInstance` 模块级变量，加载一次全局复用
- **优雅降级**：`loadFailed` 标记确保加载失败后不再重试，`available` 属性让上层感知可用性

### 5.4.3 双表存储结构

向量记忆使用两张表协同存储，通过 `rowid` 关联：

```
┌────────────────────────────────┐     ┌─────────────────────────────────┐
│         memories (普通表)       │     │    vec_memories (vec0 虚拟表)    │
├────────────────────────────────┤     ├─────────────────────────────────┤
│ id (PK, AUTOINCREMENT)        │────►│ rowid                           │
│ user_id      TEXT              │     │ embedding  float[384]           │
│ group_id     TEXT              │     │                                 │
│ scope_key    TEXT              │     └─────────────────────────────────┘
│ content      TEXT              │
│ metadata     TEXT (JSON)       │
│ created_at   TEXT              │
└────────────────────────────────┘
```

**为什么分两张表？**

1. **sqlite-vec 的 `vec0` 虚拟表只能存向量**：不支持附加文本列（rowid + embedding 就是全部）
2. **关注点分离**：`memories` 负责文本内容与元数据；`vec_memories` 只负责向量检索
3. **查询流程**：先在 `vec_memories` 做向量搜索得到 rowid → 再用 rowid 去 `memories` 取文本内容

### 5.4.4 记忆提取流程

每次 Agent 任务完成后，异步触发记忆提取：

```typescript
// src/index.ts — 任务完成后
if (config.memory.enabled) {
  const lastUserMsg = [...task.messages].reverse().find((m) => m.role === 'user');
  if (lastUserMsg) {
    void memoryStore.extractMemory(  // void = fire-and-forget
      task.userId,
      task.groupId,
      lastUserMsg.content,
      result.response.content,
    );
  }
}
```

`extractMemory()` 内部使用 Claude Haiku 从对话中提取事实：

```
┌─────────────────────────────────────────────────────────────┐
│                     extractMemory()                         │
│                                                             │
│  1. 拼接 prompt:  User: {用户消息}                          │
│                   Assistant: {助手回复}                      │
│                                                             │
│  2. 调用 Claude Haiku:                                      │
│     system = EXTRACTION_SYSTEM (提取规则)                    │
│     temperature = 0.2 (低随机性)                             │
│     maxTokens = 512                                          │
│                                                             │
│  3. 解析响应:                                                │
│     "NONE" → 无有价值信息，直接返回                           │
│     "- fact1\n- fact2" → 逐条存储                            │
│                                                             │
│  4. 对每条事实调用 storeMemory()                              │
└─────────────────────────────────────────────────────────────┘
```

**提取规则（EXTRACTION_SYSTEM prompt）**：

| 规则                 | 说明                                              |
| -------------------- | ------------------------------------------------- |
| 提取用户偏好与习惯   | 如"用户偏好 TypeScript"、"用户使用 pnpm"           |
| 提取项目技术细节     | 如"用户的项目使用 Vitest 测试"                     |
| 提取重要决策与约束   | 如"项目不使用 ORM"                                 |
| 提取个人上下文       | 如"用户希望被称为老板"                              |
| 同语言输出           | 用户说中文则中文输出，说英文则英文输出              |
| 以"User"/"用户"开头  | 明确事实主语，避免歧义                              |
| 返回 "NONE" 表示无   | 简单问候、无持久价值的对话不提取                    |

### 5.4.5 记忆存储与去重

`storeMemory()` 是核心写入方法，包含三步：

```typescript
// src/memory-store.ts
private async storeMemory(userId, groupId, scopeKey, content): Promise<void> {
  // Step 1: 文本 → 向量
  const embedding = await this.embedder.embed(content);

  // Step 2: 去重检查（相似度 ≥ 0.85 视为重复）
  const duplicates = this.db.searchMemoryVectors(embedding, 3);
  for (const { rowid, distance } of duplicates) {
    const similarity = 1 - (distance * distance) / 2;
    if (similarity >= 0.85) {
      const existing = this.db.getMemoryById(rowid);
      if (existing && existing.scope_key === scopeKey) {
        return; // 同 scope 下已有语义相近的记忆，跳过
      }
    }
  }

  // Step 3: 双表写入
  const rowid = this.db.insertMemory({ userId, groupId, scopeKey, content });
  this.db.insertMemoryVector(rowid, embedding);
}
```

**去重机制详解**：

```
新事实: "用户喜欢用 Vim 编辑器"
                │
                ▼
         embed() → [0.12, -0.34, 0.56, ...]  (384维向量)
                │
                ▼
    searchMemoryVectors(embedding, 3)  ← 取最近 3 个邻居
                │
                ▼
    ┌─ distance → similarity 转换 ──────────────────────┐
    │  similarity = 1 - (distance² / 2)                  │
    │                                                     │
    │  已有记忆: "用户偏好使用 Vim"                        │
    │  distance = 0.38  →  similarity = 0.93  ≥ 0.85    │
    │  且 scope_key 相同                                  │
    │  → 判定为重复，跳过存储                              │
    └─────────────────────────────────────────────────────┘
```

### 5.4.6 记忆召回流程

用户发送新消息时，`recall()` 方法检索相关的长期记忆：

```typescript
// src/memory-store.ts
async recall(userId, groupId, queryText): Promise<string[]> {
  const scopeKey = groupId ?? userId;
  const queryEmbedding = await this.embedder.embed(queryText);

  // 多取一些候选，用于后续 scope 过滤
  const results = this.db.searchMemoryVectors(
    queryEmbedding,
    this.config.maxRecallCount * 3,
  );

  const memories: string[] = [];
  for (const { rowid, distance } of results) {
    const similarity = 1 - (distance * distance) / 2;
    if (similarity < this.config.similarityThreshold) continue;  // 相似度不够

    const memory = this.db.getMemoryById(rowid);
    if (!memory) continue;
    if (memory.scope_key !== scopeKey) continue;  // scope 隔离

    memories.push(memory.content);
    if (memories.length >= this.config.maxRecallCount) break;
  }

  return memories;
}
```

| 参数                  | 默认值 | 说明                                 |
| --------------------- | ------ | ------------------------------------ |
| `similarityThreshold` | 0.7    | 低于此相似度的记忆不召回             |
| `maxRecallCount`      | 5      | 最多召回条数                         |

**召回流程图**：

```
用户消息: "帮我写一个 React 组件"
         │
         ▼
    embed("帮我写一个 React 组件")
         │
         ▼
    vec_memories KNN 搜索 (取 15 条候选)
         │
         ▼
    ┌─ 逐条过滤 ────────────────────────────┐
    │  1. similarity ≥ 0.7?      → 否则跳过 │
    │  2. scope_key 匹配?        → 否则跳过 │
    │  3. 数量 < maxRecallCount?  → 否则停止 │
    └───────────────────────────────────────┘
         │
         ▼
    匹配结果:
    - "用户偏好使用 TypeScript"
    - "用户的项目使用 Tailwind CSS"
```

### 5.4.7 记忆注入上下文

`SessionManager` 将召回的记忆以 system 消息形式注入：

```typescript
// src/session-manager.ts
if (this.memoryStore && messages.length > 0) {
  const lastUserMsg = [...messages].reverse().find((m) => m.role === 'user');
  if (lastUserMsg) {
    const memories = await this.memoryStore.recall(userId, groupId, lastUserMsg.content);
    if (memories.length > 0) {
      const memoryMsg: Message = {
        role: 'system',
        content: this.memoryStore.formatMemoriesMessage(memories),
        // 输出格式: "[Relevant Memories]\n- fact1\n- fact2"
      };
      // 插入到摘要之后、对话消息之前
      const summaryIdx = messages.findIndex(
        (m) => m.role === 'system' && m.content.startsWith('[Conversation Summary]'),
      );
      const insertAt = summaryIdx >= 0 ? summaryIdx + 1 : 0;
      messages.splice(insertAt, 0, memoryMsg);
    }
  }
}
```


## 5.6 成本控制

### 5.6.1 成本估算

| 操作             | 模型/组件                  | 成本                  |
| ---------------- | -------------------------- | --------------------- |
| 记忆提取         | Claude Haiku               | ~$0.001–0.005/次      |
| 上下文摘要       | Claude Haiku               | ~$0.001–0.005/次      |
| 文本嵌入         | all-MiniLM-L6-v2 (本地)   | $0（零 API 成本）     |
| 向量搜索         | sqlite-vec (本地)          | $0（纯 CPU 计算）     |

### 5.6.2 成本追踪

所有 LLM 调用（记忆提取、摘要生成）都记录到 `api_calls` 表，纳入 CostGuard 统一预算管理：

```typescript
this.db.recordApiCall({
  taskId: `memory-extract:${groupId ?? userId}`,  // 或 `summary:${sessionKey}`
  userId,
  groupId,
  provider: 'anthropic',
  model: EXTRACTION_MODEL,
  inputTokens: response.usage.inputTokens,
  outputTokens: response.usage.outputTokens,
  costUsd: cost,
});
```

### 5.6.3 关键优化

1. **本地嵌入**：使用 @huggingface/transformers 本地推理，避免每次 embedding 调用 API
2. **轻量模型**：记忆提取和摘要均使用 Claude Haiku（成本约为 Sonnet 的 1/10）
3. **去重存储**：相似度 ≥ 0.85 的重复事实不写入，控制存储膨胀
4. **增量摘要**：不重复处理已摘要的旧消息
5. **异步提取**：`void memoryStore.extractMemory()` 不阻塞用户响应

## 5.7 优雅降级

### 5.7.1 降级策略设计

记忆系统的每个组件都可能不可用（环境不支持、模型加载失败等），但**不能导致主流程崩溃**。

```typescript
// src/memory-store.ts — 构造函数
this.operational = config.enabled && db.vecAvailable && embedder.available;
```

降级矩阵：

| sqlite-vec | Embedder | config.enabled | 结果                         |
| ---------- | -------- | -------------- | ---------------------------- |
| ✅         | ✅       | ✅             | 完整长期记忆功能             |
| ❌         | ✅       | ✅             | 降级：仅短期 + 中期记忆      |
| ✅         | ❌       | ✅             | 降级：仅短期 + 中期记忆      |
| ❌         | ❌       | ✅             | 降级：仅短期 + 中期记忆      |
| —          | —        | ❌             | 记忆功能关闭                 |

### 5.7.2 各层独立降级

```
sqlite-vec 加载失败?
  → vecAvailable = false
  → vec_memories 表不创建
  → MemoryStore.operational = false
  → recall() / extractMemory() 直接返回空

Embedder 加载失败?
  → loadFailed = true
  → embedder.available = false
  → MemoryStore.operational = false
  → 同上

ContextCompressor 失败?
  → 降级为截断（只保留最近 6 条）
  → 主流程不受影响
```

## 5.8 自动清理

### 5.8.1 清理策略

| 数据类型     | 保留期 | 清理方式                               |
| ------------ | ------ | -------------------------------------- |
| messages     | 90 天  | 直接 DELETE                            |
| memories     | 365 天 | 先删 vec_memories 对应行，再删 memories |
| api_calls    | 365 天 | 直接 DELETE                            |

### 5.8.2 向量表级联删除

由于 `vec0` 虚拟表不支持外键约束，需要手动级联：

```typescript
// src/db.ts — cleanupOldData()
if (this.vecAvailable) {
  const oldMemories = this.db
    .prepare("SELECT id FROM memories WHERE created_at < datetime('now', '-365 days')")
    .all();

  if (oldMemories.length > 0) {
    for (const { id } of oldMemories) {
      this.db.prepare('DELETE FROM vec_memories WHERE rowid = ?').run(BigInt(id));
    }
    this.db
      .prepare("DELETE FROM memories WHERE created_at < datetime('now', '-365 days')")
      .run();
  }
}
```

先删向量表、再删主表，整个过程在一个事务中执行，保证原子性。
